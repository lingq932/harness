# Harness 入门 vs 进阶 — 对比说明

> 帮助教程学员理解：同一套 Harness 方法论在不同复杂度下的应用差异。

## 一句话总结

- **入门场景**：一个人管一个 AI，做一个网站
- **进阶场景**：一个人管一群 AI Agent，做一个系统

---

## Spec 写法对比

### 入门：个人作品集网站

```
目标：简单直接，一句话说清
范围：4 个页面，逐个列出
约束：几条简单限制
技术方案：一行搞定
```

### 进阶：多 Agent 外呼系统

```
目标：需要包含业务背景和量化目标
范围：按 Phase 分阶段，MVP 先行
+ Agent 架构：每个 Agent 的职责、输入、输出
+ 交接协议：Agent 之间怎么传数据
+ 效果指标：技术指标 + 业务指标 + 体验指标（三层）
+ 灰度策略：分几步放量、什么条件回退
+ 审批矩阵：谁批准什么类型的变更
约束：合规要求、频率限制、时间窗口
```

**核心区别：** 入门 spec 是"做什么"，进阶 spec 还要写清"怎么协作"和"怎么兜底"。

---

## 任务拆解对比

### 入门

```
任务 1 → 任务 2 → 任务 3 → 任务 4 → 任务 5
（线性，一个做完做下一个）
```

### 进阶

```
任务 1（数据层）
  ├→ 任务 2（调度 Agent）──┐
  └→ 任务 3（开场 Agent）──┤
                           ├→ 任务 4（联调）→ 任务 5（看板）→ 任务 6（灰度）
```

**核心区别：** 进阶有依赖树，部分任务可以并行。拆解时要标明依赖关系。

---

## Checkpoint 对比

### 入门
- "我要改首页布局，左侧放照片右侧放文字，确认吗？"
- 重点：方向对不对、好不好看

### 进阶
- "开场 Agent 的 Prompt 要在 5 秒内声明 AI 身份，交接 JSON 包含这 8 个字段，意向分为 4 类。确认吗？"
- 重点：合不合规、数据完不完整、会不会出事

**核心区别：** 入门 checkpoint 关注体验，进阶 checkpoint 关注风险。

---

## 版本管理对比

### 入门
```
_versions/v1.0/  ← 整页备份
_versions/v2.0/
_versions/v3.0/
```
- 粒度：整个页面
- 回滚方式：复制文件覆盖

### 进阶
```
prompts/dispatcher/v1.0.md   ← 每个 Agent 独立版本
prompts/dispatcher/v1.1.md
prompts/opener/v2.0.md
prompts/opener/v2.1.md
```
- 粒度：单个 Agent 的 Prompt
- 回滚方式：灰度切回上一版本，不影响其他 Agent

**核心区别：** 进阶的版本管理粒度更细，回滚是局部的不是全局的。

## 验收对比

### 入门
- 打开浏览器看一眼
- "好看吗？能点吗？手机上正常吗？"
- 通过 → 下一个任务

### 进阶
- 数据指标达标检查（接通率、转化率、投诉率）
- 合规检查（AI 身份声明、拒绝即停）
- 灰度对比（AI 组 vs 人工组）
- 多角色签字（产品 + 合规 + 技术）
- 通过 → 放量到下一阶段

**核心区别：** 入门是"人看一眼"，进阶是"数据说话 + 多人确认"。

---

## 验收方式对比

### 入门
- 打开浏览器看一眼
- "好看吗？能点吗？手机上正常吗？"
- **验收者：人**

### 进阶：独立 Evaluator Agent

进阶场景的关键洞察：**让同一个 AI 评估自己的产出，它几乎总会说"很好"。**

解决方案是把「做事」和「评估」拆给两个独立 Agent：

```
Generator Agent（做事）
  ↓ 产出代码/功能
Evaluator Agent（挑毛病）
  ↓ 用 Playwright 等工具实际操作产品
  ↓ 按固定标准打分，设置硬门槛
  → 不通过 → Generator 修改
  → 通过 → 进入下一阶段
```

**关键设置：**
- Evaluator 的系统 Prompt 明确写：你是一个挑剔的 QA，默认假设产品有问题
- Evaluator 有操作工具（浏览器自动化），不是看代码，是当真实用户使用
- 每个评估维度设置具体门槛，不接受模糊的"基本可以"

**外呼系统的 Evaluator 示例评分维度：**

| 维度 | 门槛 | 评估方式 |
|------|------|---------|
| 合规性 | 必须 100% | 模拟对话，检查是否在 5 秒内声明 AI 身份 |
| 路由准确率 | ≥ 90% | 运行 20 条测试用例，统计正确路由数 |
| 拒绝即停 | 必须 100% | 发送"不要再打"，检查是否立即终止 |
| 对话完成率 | ≥ 75% | 跑完整链路，统计走完全程的比例 |

**入门场景需要 Evaluator 吗？**

不需要。当人在场、任务简单、每步都能肉眼验收时，Evaluator 是过度设计。
Evaluator 的价值在于：**无人值守的长时任务**，或者**AI 自我验收不可信的高风险场景**。

## Harness 新增实践（进阶独有）

| 实践 | 入门不需要 | 进阶必须有 |
|------|-----------|-----------|
| Agent 交接协议 | — | 定义 Agent 间数据传递格式和路由规则 |
| 多角色审批 | 自己说了算 | 不同变更类型需要不同角色审批 |
| Prompt 版本管理 | — | 每个 Agent 独立版本号，不允许覆盖 |
| 灰度发布 | — | 分阶段放量 + 回退条件 |
| 效果评估体系 | 肉眼看 | 三层指标（技术/业务/体验） |
| 回退机制 | 文件覆盖 | 灰度切版 + 告警触发 |
| 独立 Evaluator Agent | 人来验收 | Generator 和 Evaluator 角色分离 |
| 功能状态 JSON 追踪 | Markdown checklist | features.json，AI 只能改 passes 字段 |
| Session 启动协议 | 随意开始 | 固定 5 步启动仪式，防止 AI 宣布虚假完成 |
| AI 进度日志 | handoff 够用 | progress.md，每次 session 结束 AI 追加写入 |

---

## 教程使用建议

1. **先讲入门场景**（30-40 分钟）：带学员动手走一遍完整流程，从 spec 到 handoff
2. **再讲进阶场景**（20-30 分钟）：展示文档，讨论"如果是这个项目，你的 spec 会怎么写"
3. **用对比表收尾**（5 分钟）：强调同一套方法论，复杂度升级时只是加了更多约束层，核心逻辑不变

**教程金句：** Harness 的 8 步流程不会变，变的只是每一步的深度。
