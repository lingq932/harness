# Harness Engineering 通用模板

> 用于约束 AI Agent 在工程项目中的行为，确保可控、可回滚、可交接。

---

## 一、核心理念

Harness 的本质是：**人控盘，AI 执行，全程有据可查。**

- AI 是非确定性的，同样的输入可能产出不同结果
- 因此需要「缰绳」：明确的规格、检查点、回滚机制
- 人是「控盘者」，负责决策和验收；AI 是「执行者」，负责产出和建议

---

## 二、三个核心文件

### 1. CLAUDE.md — AI 的「开机配置」

AI 每次进入新 session 时第一个读的文件。它不是需求文档，是**行为约束**——告诉 AI 这是什么项目、当前在哪个阶段、有哪些不能做的事。

```markdown
# 项目名 — CLAUDE.md

## 项目简介
一句话：这是什么、给谁用。

## 当前阶段
第几版 / 当前在做什么任务块。

## 关键约束（必读，放最前面）
- 不允许自行添加 spec 之外的功能
- 破坏性操作（删文件、推代码、发请求）执行前必须先确认
- [项目特定约束，例如：不改动 data/ 目录的现有文件]

## 技术栈
- 框架 / 语言 / 部署方式（一行）

## 重要文件
- `_harness/spec.md` — 需求规格（完整版）
- `_harness/handoff.md` — 项目状态与交接

## 禁止行为
- 不在没有 checkpoint 的情况下跨任务块执行
- 不修改 spec 里标注「暂不处理」的功能
```

**使用规则：**
- 每个 session 开始时 AI 先读这个文件，再读 spec
- 关键约束放第一屏（AI 注意力 U 型曲线，开头最稳）
- 有新的重要约束发现时，当场补进来，不要等到写 handoff 才整理

### 2. spec.md — 单一事实来源

```markdown
# 项目名 — Spec

## 目标
一句话说清楚：做什么、给谁用、核心价值。

## 范围
### 包含
- 功能模块 1：描述
- 功能模块 2：描述
- ...

### 视觉/交互要求
- 风格方向
- 关键约束

## 技术方案
- 框架 / 语言 / 部署方式

## 约束
- 不做什么（明确边界）
- 数据来源限制
- 性能要求

## 暂不处理项
- 明确列出推迟的功能，防止范围蔓延
```

**使用规则：**
- Spec 是唯一的需求来源，AI 不应自行添加 spec 之外的功能
- 任何需求变更必须先更新 spec，再执行
- Spec 由人审批，AI 只能建议修改

### 3. handoff.md — 交接文档

```markdown
# 项目名 — Handoff

## 项目状态
当前阶段 + 线上地址 + 如何运行

## Spec 完成度
表格形式逐项对照

## 文件结构
关键文件说明

## 如何继续迭代
常见操作的具体步骤

## 可继续优化的方向
checklist 形式列出
```

**使用规则：**
- 项目阶段性完成时编写
- 确保任何人（或另一个 AI session）拿到 handoff 就能接手

## 三、8 步工作流 SOP

```
Step 1: 明确目标 → 人描述意图，AI 复述确认
Step 2: 编写 Spec → AI 起草，人审批
Step 3: 任务拆解 → AI 拆成可执行的任务块，人确认顺序
Step 4: Checkpoint → 每个任务块执行前，AI 陈述要做什么、预期产出
Step 5: 执行 → AI 动手，人观察
Step 6: 验收 → 人检查产出，通过则进入下一块
Step 7: 迭代 → 不满意则调整，可回滚到上一版本
Step 8: 交接 → 编写 handoff.md，项目可被任何人接手
```

**关键原则：**
- **一次只做一块**，没有批准不跳到下一块
- **Checkpoint before Execute**：先说再做，不要闷头干
- **人批准才推进**：AI 完成后等待确认，不自动继续

## 四、3 层目标管理

| 层级 | 内容 | 谁管 |
|------|------|------|
| 战略层 | 项目目标、核心价值 | 人定义 |
| 战术层 | 任务拆解、优先级排序 | AI 建议，人审批 |
| 执行层 | 具体代码、文件修改 | AI 执行，人验收 |

AI 不应在执行层擅自做战略层的决策（比如自行添加功能）。

---

## 五、5 个漂移信号

当出现以下情况时，立即暂停并与人确认：

| 信号 | 表现 | 应对 |
|------|------|------|
| 范围蔓延 | AI 开始做 spec 之外的事 | 停下来，问人是否要扩展 spec |
| 技术偏离 | 实现方式与 spec 不符 | 回到 spec 确认技术方案 |
| 质量下降 | 产出明显粗糙或有 bug | 先修复再继续，不要带病推进 |
| 沟通断裂 | 人和 AI 理解不一致 | 复述理解，对齐后再继续 |
| 循环漂移 | 同一问题尝试超 3 次仍未解决 | 立即停止，人重新描述需求或换思路 |

## 六、版本管理与回滚

```
src/_versions/
├── v1.0/    ← 每次大改前备份
├── v2.0/
└── v3.0/
```

**规则：**
- 每次视觉大改或结构变更前，先备份当前版本
- 命名清晰（v1.0、v2.0），备份目录里放关键文件
- 不满意时可以一键回滚：复制备份文件覆盖当前

## 七、Eval & Cost 检查

### Eval：轻量回归清单

不需要自动化。在 `_harness/` 目录里建一个 `eval-scenarios.md`，列出 5-10 个「这个项目下应该能做对的典型场景」，每次大改后手动跑一遍对比。

```markdown
# 项目名 — Eval Scenarios

## 场景 1：[场景名]
- 操作：...
- 预期结果：...
- 上次验证：YYYY-MM-DD ✅ / ❌

## 场景 2：...
```

**触发时机：**
- 修改 CLAUDE.md 或 spec 之后
- 切换底层模型或 API 版本之后
- 完成一个大任务块之后

### Cost：Token 消耗意识

AI 任务的成本不是「按次收费」，而是**按 token 消耗**——一个失控的 sub-agent 死循环可以在 10 分钟内烧掉正常任务 100 倍的成本。

**日常检查习惯：**
- 单次任务结束后，感知一下对话轮数是否异常多（超过 20 轮要想想为什么）
- 遇到 AI 反复修改同一文件、反复尝试同一方案 → 即是「循环漂移」信号，同时也是 cost 失控信号
- 复杂项目建议定期看 API 用量面板，识别哪类任务消耗最高

**高 cost 风险场景（重点注意）：**
| 场景 | 风险原因 | 应对 |
|------|----------|------|
| Sub-agent 派发 | 每个子任务独立计费，失控时叠加 | 给每个 sub-agent 明确 max 步数 |
| 大文件塞进 context | 100K token 的文档每次都全量传 | 用 RAG 或摘要替代，只传相关片段 |
| 反复迭代同一任务 | 每轮修改都消耗完整 context | 循环漂移信号出现即停，重新对齐再继续 |
| 模型升级未测试 | 新模型可能更啰嗦，token 倍增 | 切模型前跑 eval 场景对比输出长度 |

## 八、角色设置（可选）

在教学或复杂项目中，可以让 AI 扮演双重角色：

- **AI 角色**：执行任务、回答问题、产出代码
- **教练角色**：解释为什么这么做、点评 Harness 方法论的应用

默认对话对象是 AI，除非人明确说"问教练"。

---

## 九、Quick Start Checklist

开始一个新的 Harness 项目时：

- [ ] 创建 `_harness/` 目录
- [ ] 人描述目标，AI 复述确认
- [ ] AI 起草 `spec.md`，人审批
- [ ] AI 生成 `CLAUDE.md`（项目简介 + 关键约束），人确认后放项目根目录
- [ ] AI 拆解任务块，人确认
- [ ] 逐块执行：checkpoint → 执行 → 验收
- [ ] 每次大改前备份版本
- [ ] 建 `_harness/eval-scenarios.md`，列出 5-10 个典型验收场景
- [ ] 完成后编写 `handoff.md`，同步更新 `CLAUDE.md` 的当前阶段
